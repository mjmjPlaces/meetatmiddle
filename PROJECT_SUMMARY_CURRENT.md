# 📍 프로젝트: 딱중간 (Midpoint Navigator)

> **컨셉**: 수학적 정중앙이 아니라, 대중교통/상권/합의 가능성을 함께 고려한 “사회적 최적” 모임 지점을 제안

## 1) 현재 구현된 핵심 기능

### ⚖️ 탐색 모드
- **균등 모드 (`balanced`)**
  - 평균/편차/최대시간 기반으로 특정 개인의 과도한 희생을 줄이는 방향
- **차등 모드 (`majority`)**
  - 원거리 일부의 영향력을 완화해 다수의 총 효율을 높이는 방향

### 🚉 후보지 선정 로직
- 친구 출발지 지오코딩 후 중심점 계산
- 중심 반경 내 지하철역 후보 탐색(Kakao)
- 직선거리 기반 1차 필터링(Pruning) 후 정밀 평가(ODsay)

### 🏙 거점역 마스터(Tier) 반영
- `station_master` 리스트를 기반으로 **검증된 약속 장소** 우선
- Tier(1/2/3) 보정 점수 적용
- 비우선 역은 패널티, 일부는 지정 거점역으로 **강제 Shift** 적용
- 결과 UI에 Tier 및 후보 보정 정보 노출

### 🧭 상세 경로/지도 UX
- 결과는 Top1 중심으로 표시
- 지도 마커 + “선정 이유” 배지
- 세부내용 조회 시:
  1. 친구 전체 소요시간 요약
  2. 선택 친구의 구간별 대중교통 경로 상세
- 친구별 경로 지도 오버레이(노선 색상/출발·도착 마커)

### 🔗 공유
- 카카오 공유 버튼으로 결과 요약(지점/이유/평균·최대시간/지도 링크) 전송

## 2) 운영/성능 대비 상태

### ⚡ 최적화
- In-memory TTL 캐시(지오코딩/경로/차선 정보)
- in-flight dedupe(중복 외부 호출 방지)
- ODsay 호출 예산(max call budget) 및 rate limiting

### 🛡 운영 안정성
- CORS 화이트리스트(`ALLOWED_ORIGINS`)
- 운영 메트릭 API: `/api/ops/metrics`
- ODsay 일 예산 90% 임계시 degraded mode 자동 진입

## 3) 배포 구조(현재 권장)

- **Frontend**: Vercel
- **Backend**: Railway (Node.js/Express)
- 외부 API: Kakao Local / Kakao Maps JS / ODsay

## 4) 아직 계획 단계(다음 과제)

- [ ] 심리 가중치 수식(서울↔경기 저항, 한강 장벽 등) 정식 도입
- [ ] 50분 초과 누진 페널티 함수 도입
- [ ] 상권 밀도 실시간 점수화(Kakao 카테고리 집계)
- [ ] Redis 캐시 도입(재시작 내구성/운영 트래픽 대응)
- [ ] 30개 거점역 리스트 → 100개 이상 확장 및 Shift 맵 정교화

## 5) 수익화 방향 (초안)

- Affiliate: 추천 지점 인근 예약/티켓 링크 연동
- B2B API: 모임/캘린더 서비스에 추천 엔진 제공
- 지역 인사이트: 익명 집계 기반 시간/거점 분석 리포트
